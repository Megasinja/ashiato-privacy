# ASHIATO 軌跡再生：適応カメラ（自動ズーム・自動センタリング）設計メモ

検討日: 2026-06-14

関連: `android-dev-summary.md` §2（再生）、プロトタイプ `trajectory.html`。

---

## 1. 目的

軌跡を再生するとき、**再生時刻まわりの移動速度に応じて地図のズーム率と中心座標を
自動調整**し、常に見やすい画にする。

直感:
- **遅い（停止・徒歩）** → ズームイン。細かい道や立ち寄りが見える。
- **速い（車・電車）** → ズームアウト。マーカーが画面外に飛ばず、進行先も見える。
- **進行方向の先**を少し多めに映す（来た道より、これから行く道を見せる）。
- **カーブ・密な区間**は寄り、**直線の高速移動**は引く。

NG例: マーカー位置に固定ズームで張り付くと、高速移動でビュンビュン流れて酔うか、
逆に徒歩でスカスカの画になる。

---

## 2. 2つの実装アプローチ

### アプローチB（推奨）：再生時刻まわりの「窓」を fitBounds する

ユーザーの「**再生時間周辺**の移動速度に合わせて」という発想に最も素直。

1. 現在の再生時刻 `t` の前後に**時間窓**（または距離窓）を取る。
2. その窓に入る軌跡の点群の**バウンディングボックス**を計算。
3. カメラをそのボックスに **fitBounds**（パディング付き）する。
4. フレーム間で**スムージング**して滑らかに追従。

これだけで **ズームも中心も自動で決まる**:
- 高速移動 → 窓内の移動距離が大きい → ボックスが広い → **自動でズームアウト**。
- 低速・カーブ → ボックスが小さい/密 → **自動でズームイン**。
- 窓を「後ろより前に厚く」取れば、自然に**進行方向先読み**になる。

```kotlin
// 窓は速度で可変（速いほど前を広く見る）。back:ahead = 30:70 で先読み寄り。
fun cameraTargetAt(t: Long): CameraTarget {
    val v = smoothedSpeedAt(t)                  // m/s（§3）
    val windowSec = windowSeconds(v)            // 例 6〜40秒（§5表）
    val back  = windowSec * 0.30
    val ahead = windowSec * 0.70
    val seg = pointsInTimeRange(t - back.sec, t + ahead.sec) // 距離モードは距離窓に
    val box = boundingBox(seg).extend(markerPosAt(t))        // マーカーも必ず含める
    return fitBounds(box, paddingPx = 80, maxZoom = 18f)     // 地図SDKの計算を利用
}
```

### アプローチA：速度→ズーム関数 ＋ 先読みセンター（明示制御）

ボックスではなく、速度から**直接ズーム値**を、進行方向から**中心**を計算する。
細かく効かせたいとき向け。Bと併用も可（Bで決めた値をAで微調整）。

```kotlin
// 速度→ズーム（連続・対数寄り。4km/hで17、90km/hで12へなだらかに）
fun zoomForSpeed(kmh: Double): Float {
    val z = 17.0 - 5.0 * (ln(kmh + 1) - ln(5.0)) / (ln(91.0) - ln(5.0))
    return z.coerceIn(12.0, 18.0).toFloat()
}

// 先読み中心：マーカーから進行方向へ「数秒ぶん」前に置く
fun lookAheadCenter(t: Long): LatLng {
    val pos = markerPosAt(t)
    val headingRad = smoothedHeadingAt(t)            // §4
    val aheadMeters = smoothedSpeedAt(t) * LOOK_AHEAD_SEC  // 例 4秒ぶん
    return projectAlong(pos, headingRad, aheadMeters)
}
```

> **推奨**: まず B（fitBounds 窓）で土台を作り、必要なら A の先読み/微調整を重ねる。
> B は地図SDKの `fitBounds` をそのまま使えるので実装が軽く、破綻しにくい。

## 3. 速度の推定（瞬時値を使わない）

GPSの瞬時速度はノイズが多い。**再生時刻まわりの窓で平均**した滑らかな速度を使う。

```kotlin
// t を中心に ±W 秒の区間の平均速度（距離 / 時間）。EXIF speed があれば併用可。
fun smoothedSpeedAt(t: Long, halfWindowSec: Double = 4.0): Double {
    val a = pointAtOrBefore(t - halfWindowSec.sec)
    val b = pointAtOrAfter (t + halfWindowSec.sec)
    val dist = pathDistanceBetween(a, b)            // 累積距離の差（前処理済み）
    val dt   = (b.timestamp - a.timestamp) / 1000.0
    return if (dt > 0) dist / dt else 0.0
}
```

さらにフレーム間で指数平滑（§6）をかけ、ズーム値のチラつきを防ぐ。

## 4. 進行方向（heading）

先読みセンターやカーブ判定に使う。**少し離れた前後の点**から方位を出し、平滑化する。

```kotlin
fun smoothedHeadingAt(t: Long): Double {
    val p0 = markerPosAt(t)
    val p1 = markerPosAt(t + 2_000)   // 2秒先の位置
    return bearing(p0, p1)            // ラジアン。停止中は前回値を保持
}
```

停止中（速度≒0）は方位が暴れるので、**前回の方位を保持**する。

## 5. カーブ・密度への対応

直線高速は引き、カーブや立ち寄りは寄る。アプローチBなら**窓内の点が作るボックスの
形**で自動的にそうなるが、明示的に効かせたい場合の指標:

- **方位変化量**: 窓内の heading の総変化（積算 |Δbearing|）が大きい＝カーブ多い → ズーム+1。
- **点の散らばり**: 窓内点の重心からの分散が小さい＝うろうろ/立ち寄り → ズームイン。

### パラメータ目安

| 状況 | 速度 | 時間窓(秒) | 目安ズーム | 先読み |
|------|------|-----------|-----------|--------|
| 停止・立ち寄り | 〜1 km/h | 6 | 17〜18 | ほぼ無し |
| 徒歩 | 〜5 km/h | 8 | 16〜17 | 弱 |
| 自転車 | 〜20 km/h | 14 | 15 | 中 |
| 車（市街） | 〜50 km/h | 25 | 13.5 | 強 |
| 高速・電車 | 60 km/h〜 | 40 | 12 | 強 |

（窓秒・ズームは連続関数で補間。表は折れ点の目安。）

## 6. スムージング（最重要）

ズームと中心が毎フレーム飛ぶと**見るに耐えない**。必ず**目標値へ徐々に寄せる**。

```kotlin
// 各フレーム：現在カメラ → 目標カメラ へ指数補間（alphaは0.06〜0.12）
fun stepCamera(target: CameraTarget, dt: Double) {
    val a = 1 - exp(-dt / TAU)                 // 時間ベースで端末FPSに依存しない
    camZoom   += (target.zoom - camZoom) * a
    camCenter  = lerp(camCenter, target.center, a)
    // ズーム変化率の上限（急なズームを抑制）
    camZoom = clampRate(camZoom, prevZoom, maxZoomPerSec * dt)
    map.moveCamera(camCenter, camZoom)
}
```

- `TAU`（時定数）を 0.5〜1.0 秒に。大きいほどゆったり。
- **ヒステリシス**: 速度がズーム閾値付近を行き来してもズームが振動しないよう、
  バンド切替に余裕（例: 上げは20km/h、下げは15km/hで戻す）を持たせる。
- 中心は marker への追従と先読みオフセットの**合成**を平滑化する。

## 7. 2つの再生モードとの関係

`android-dev-summary.md` §2 の2モードと独立に働く（速度は実データ由来のため）:

- **時間速度一定モード**: 画面内のマーカー速度が実速度に比例して変動する。
  → 適応カメラの効果が**最も大きい**（高速区間で自動ズームアウトが効く）。
- **移動速度一定モード**: マーカーは画面上で一定速度。
  → カメラは安定方向。ただし「実速度が速い区間＝引く」を効かせると、
    その区間に来たことが画でも伝わり情報量が増す。窓は**距離窓**で取ると自然。

---

## 8. ユーザー操作との両立

- **適応カメラ ON/OFF トグル**を用意（固定ズーム派のため）。
- **手動操作中は自動追従を一時停止**：ユーザーがドラッグ/ピンチしたら追従OFF。
  （プロトタイプ `trajectory.html` は既に `dragstart` で `followMap=false`。
   同じ仕組みにズーム自動制御も連動させる。）
- 「現在地に戻る📍」で追従と適応カメラを再開。
- 再生停止中はカメラを動かさない。

## 9. パラメータ一覧（チューニング用）

| パラメータ | 役割 | 目安 |
|-----------|------|------|
| `windowSeconds(v)` | 再生時刻まわりの窓幅 | 6〜40秒（速度で可変） |
| back:ahead 比 | 後ろ/前の配分（先読み量） | 30:70 |
| `paddingPx` | fitBounds の余白 | 60〜100px |
| `maxZoom` / `minZoom` | ズーム上下限 | 18 / 11 |
| `LOOK_AHEAD_SEC` | 先読み秒数（アプローチA） | 3〜6秒 |
| `TAU` | カメラ平滑の時定数 | 0.5〜1.0秒 |
| `maxZoomPerSec` | ズーム変化率上限 | 0.8〜1.5 |
| ヒステリシス幅 | ズームバンド切替の余裕 | 速度で±20% |

## 10. エッジケース

- **データが疎**（点が少ない/間隔が大きい）: 窓内に点が1つしか無いと box が潰れる。
  → マーカー位置を中心に**最小ズーム上限**でフォールバック。
- **巨大ジャンプ**（GPS外れ値/トンネル復帰）: 窓に飛び値が入ると box が暴れる。
  → §battery 4-1 の外れ値除去後の系列を使う。表示用に上限ズーム変化率で吸収。
- **シーク（スライダー大移動）**: 連続平滑だと飛び先まで間延びする。
  → シーク直後は1回だけ**スナップ**（補間せず即セット）してから平滑再開。
- **停止が長い**: 窓が点に潰れる → 最大ズームで張り付き。立ち寄りとして自然なのでOK。

## 11. プロトタイプ（trajectory.html）への展開と検証

現状の `trajectory.html` は `followMap` 時に `map.panTo(markerPos)` するのみ（ズーム固定）。
本構想を入れるなら:

1. 毎フレームの `render()` 末で `cameraTargetAt(currentTime)` を計算（アプローチB）。
2. `stepCamera()` で `map.setView(center, zoom, {animate:false})` を平滑更新。
3. ヘッダーに「適応カメラ ON/OFF」トグルを追加（既存の📍追従と併存）。

検証:
- 徒歩→自転車→車のように速度が変わるサンプル（`loadDemo()` を拡張）で、
  速い区間で引き・遅い区間で寄るか、カクつかないかを目視。
- シーク直後にスナップして破綻しないか。
- 手動ドラッグで追従が切れ、📍で復帰するか。
